# 🐞 Трекер Задач и Ошибок Проекта

**Версия:** 1.0
**Дата последнего обновления:** 2025-09-05

Этот документ является центральным местом для отслеживания всех известных проблем, предложений по улучшению и истории их решения.

## 1. Открытые Проблемы и Предложения

Здесь перечислены задачи, которые находятся в работе или ожидают реализации.

### `ISSUE-001`: Улучшить систему логирования
*   **Описание:** Текущая система логирования функциональна, но не имеет четкой структуры. Сложно фильтровать ошибки по уровню критичности или по модулю системы.
*   **Предложение:** Внедрить структурированное логирование (например, в формате JSON), добавить контекст в логи (например, `user_id`).
*   **Статус:** 🟡 Открыто

### `ISSUE-002`: Возможные утечки памяти
*   **Описание:** Есть подозрение, что при длительной работе (несколько дней без перезапуска) бот начинает потреблять больше памяти. Точная причина не установлена.
*   **Предложение:** Провести профилирование памяти, использовать инструменты для обнаружения утечек (например, `tracemalloc`).
*   **Статус:** 🟡 Открыто

### `ISSUE-003`: Конфликты при запуске нескольких экземпляров
*   **Описание:** Хотя есть система блокировки (`.lock` файл), она не всегда надежно работает, что может приводить к запуску двух копий бота одновременно.
*   **Предложение:** Улучшить механизм блокировки, возможно, используя системные средства, а не просто файл.
*   **Статус:** 🟡 Открыто

### `ISSUE-004`: Отсутствие автоматических тестов
*   **Описание:** В проекте полностью отсутствуют автоматические тесты, что делает каждое изменение потенциально опасным.
*   **Предложение:** Начать покрытие кода Unit-тестами, в первую очередь для критических модулей (`auth_cache`, `sheets_client`, `promotions_client`).
*   **Статус:** 🟡 Открыто

## 2. Архив Решенных Проблем

Здесь задокументированы все значимые проблемы, которые уже были исправлены.

### `FIX-001`: Критическая ошибка авторизации на мобильных устройствах
*   **Проблема:** Кнопка авторизации, реализованная через `InlineKeyboardMarkup`, не работала на некоторых мобильных платформах (особенно iOS), так как `web_app` в инлайн-кнопках не всегда надежно отправляет данные (`window.Telegram.WebApp.sendData`).
*   **Решение:** Был внедрен гибридный подход. Для первоначальной авторизации, где требуется гарантированная отправка данных, используется `ReplyKeyboardMarkup` (кнопка под полем ввода). После успешной авторизации пользователю предоставляется постоянная клавиатура с доступом в личный кабинет уже через `InlineKeyboardMarkup`, так как там отправка данных не требуется.
*   **Статус:** ✅ Решена
*   **Источник:** `MOBILE_AUTH_DIAGNOSTIC.md`, `MOBILE_AUTH_FIX.md`

### `FIX-002`: Некорректное отображение акций в мини-приложении
*   **Проблема:** При большом количестве акций список в мини-приложении становился нечитаемым. Кнопки "Уведомить" и "Подробнее" не всегда корректно отрабатывали.
*   **Решение:** Пользовательский интерфейс был переработан с использованием "аккордеона". Каждая акция представляет собой сворачиваемую карточку. Это позволило компактно отображать список. Внутри каждой карточки теперь есть медиа-контент и кнопки, работающие в контексте этой акции.
*   **Статус:** ✅ Решена
*   **Источник:** `PROMOTIONS_FIXES_REPORT.md`, `WEBAPP_BUTTON_FIX_REPORT.md`

### `FIX-003`: Бот "замерзал" при обращении к Google Sheets
*   **Проблема:** Любые операции с Google Sheets (чтение, запись) были синхронными, что блокировало основной поток бота. Во время ожидания ответа от Google API бот не отвечал на другие команды.
*   **Решение:** Все обращения к `gspread` были обернуты в `asyncio.to_thread`, что позволяет выполнять их в отдельном потоке, не блокируя основной цикл событий. Был создан `AsyncPromotionsClient` для полной асинхронной работы с таблицей акций.
*   **Статус:** ✅ Решена
*   **Источник:** `BUG_FIXES_REPORT.md`

### `FIX-004`: Медленная проверка авторизации при каждом сообщении
*   **Проблема:** При каждом сообщении от пользователя бот делал запрос к Google Sheets, чтобы проверить статус авторизации. Это создавало лишнюю нагрузку и замедляло ответ.
*   **Решение:** Внедрен локальный кэш (`auth_cache.py`). После первой успешной авторизации ID пользователя сохраняется в кэш. Все последующие проверки происходят мгновенно через локальный кэш, и только при его отсутствии делается запрос к Google Sheets.
*   **Статус:** ✅ Решена
*   **Источник:** `SCALING_AND_CACHE_FIX.md` (упоминалось в анализе)
